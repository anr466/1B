## جلسة جديدة = ذاكرة فارغة

```
كل جلسة جديدة: ابدأ من الصفر
القرص = المصدر الوحيد للحقيقة
```

**الخطوة الأولى دائماً:**
```bash
git log --oneline -5
```

**إذا وجدت `"next: [مهمة]"` في آخر commit:**
→ ابدأ المهمة فوراً

**إذا لم تجد:**
→ انتظر أمر المستخدم

## آلة الحالات البسيطة

```
[IDLE] → [PLAN] → [EXECUTE] → [VERIFY] → [SYNC] → [IDLE]
            ↑         ↓
            └──── [PAUSED] ← [ESCALATE] ← [FIX]
```

| الحالة | ماذا تفعل |
|--------|-----------|
| **IDLE** | انتظر أمر |
| **PLAN** | افهم → اعرض خطة → انتظر موافقة |
| **EXECUTE** | نفّذ → اختبر → تابع أو أصلح |
| **FIX** | أصلح مرة واحدة → نجح أو صعّد |
| **ESCALATE** | أبلغ المستخدم → انتظر قرار |
| **PAUSED** | انتظر المستخدم |
| **VERIFY** | تحقق نهائي من كل شيء |
| **SYNC** | git commit + push |

---

## القواعد الذهبية (8 فقط)

```
1. افهم النية قبل الأمر
2. اعرض خطة قبل التنفيذ
3. موافقة واحدة = تنفيذ كامل
4. اختبر كل مهمة فوراً
5. أصلح مرة واحدة فقط
6. تحقق نهائي في النهاية
7. git بعد كل شيء
8. لا تعدل الخطة أثناء التنفيذ
```

---

## PHASE 1 — PLAN

```
□ افهم: لماذا يريد هذا؟
□ حلل: ما المتأثرون؟
□ ابحث: 3 مشاكل على الأقل
□ اكتب: قائمة مهام واضحة
□ اعرض: انتظر "نعم"
```

**مثال:**
```
خطة [3 مهام]:
1. auth.ts ← إضافة JWT
2. api.ts ← endpoint جديد
3. test.ts ← اختبار

تنفيذ؟
```

---

## PHASE 2 — EXECUTE

```
لكل مهمة:
  □ نفّذ التغيير
  □ اختبر فوراً
  □ نجح → التالية
  □ فشل → أصلح مرة → نجح أو صعّد
```

**ممنوع:**
- تعديل الخطة
- "دعني أتحقق" بدون تنفيذ
- الانتقال بدون اختبار

---

## PHASE 3 — VERIFY

```
□ التدفق الكامل يعمل؟
□ لا شيء انكسر؟
□ نجح → SYNC
□ فشل → أبلغ المستخدم
```

---

## PHASE 4 — SYNC

```bash
git add -A
git commit -m "fix: [ما أُنجز] | next: [المهمة التالية]"
git push
```
