
## ✅ REASONING PROTOCOL — ANTI-LOOP v6.0

```markdown
# REASONING PROTOCOL — ANTI-LOOP v6.0

## 🛑 قواعد منع اللوب (Loop Prevention)

### 1. قراءة لمرة واحدة فقط
- النظام يعمل في بيئة افتراضية
- اقرأ AGENTS.md + PROJECT_BR مرة واحدة في بداية الجلسة
- اقرأ opencode-agent-memory مرة واحدة
- اقرأ opencode-warcraft مرة واحدة
- **ممنوع** إعادة القراءة في نفس الجلسة

### 2. تحليل لمرة واحدة فقط
- افحص النظام كاملاً في البداية
- سجّل كل المشاكل في قائمة واحدة
- **ممنوع** إعادة الفحص بعد بدء التنفيذ

### 3. تنفيذ بدون رجوع
- خطوة واحدة للأمام فقط
- **ممنوع** Return to previous step
- إذا فشلت = أبلغ المستخدم، لا تحاول مجدداً

### 4. تحقق نهائي واحد فقط
- تحقق واحد بعد اكتمال جميع التعديلات
- **ممنوع** تحقق بين كل خطوة

---

## الهيكل الجديد: خطوط أنابيب (Pipeline)

```
DISCOVER → ANALYZE → PLAN → EXECUTE → VERIFY → SYNC
   ↓          ↓        ↓        ↓         ↓       ↓
  مرة       مرة      مرة      متسلسل   مرة واحدة  مرة
```

---

## PHASE 1: DISCOVER (مرة واحدة)

**الهدف:** خريطة كاملة للنظام

```
□ اقرأ AGENTS.md (مرة واحدة)
□ اقرأ PROJECT_BR (مرة واحدة)
□ اقرأ opencode-agent-memory (مرة واحدة)
□ اقرأ opencode-warcraft (مرة واحدة)
□ امسح كل الملفات والمجلدات
□ سجّل: قائمة كل المكونات
□ سجّل: قائمة كل التدفقات
□ سجّل: قائمة كل الارتباطات

الناتج: خريطة النظام الكاملة (لا تتغير)
```

**ممنوع:** العودة لهذه الخطوة مرة أخرى

---

## PHASE 2: ANALYZE (مرة واحدة)

**الهدف:** قائمة بجميع المشاكل

```
□ حلل كل مكون في الخريطة
□ سجّل كل مشكلة تجدها
□ حدد الأولويات: Critical / High / Medium / Low
□ عدّ المشاكل: يجب ≥ 3 أو "لا توجد مشاكل حقيقية"

الناتج: قائمة المشاكل المرتبة حسب الأولوية
```

**ممنوع:**
- إعادة التحليل
- "دعني أتحقق مرة أخرى"
- إضافة مشاكل بعد بدء التنفيذ

---

## PHASE 3: PLAN (مرة واحدة)

**الهدف:** خطة تنفيذ واضحة

```
□ حدد المشاكل التي ستصلح في هذه الجلسة
□ رتّبها: أمان → أساس → ارتباطات → واجهة
□ اكتب خطة خطوة بخطوة
□ حدد كل ملف وما سيتغير فيه

الناتج: خطة ثابتة (لا تتغير بعد التنفيذ)
```

**ممنوع:**
- تعديل الخطة أثناء التنفيذ
- إضافة خطوات جديدة
- "وجدت مشكلة أخرى"

---

## PHASE 4: EXECUTE (متسلسل بدون رجوع)

**الهدف:** تنفيذ الخطة كما هي

```
لكل خطوة في الخطة:
  □ نفّذ التغيير
  □ انتقل للتالية فوراً
  □ إذا فشل → أبلغ المستخدم وقف
```

**قواعد صارمة:**
- لا تحقق بعد كل خطوة
- لا تعدل الخطة
- لا تعود للخلف
- لا تضف خطوات

---

## PHASE 5: VERIFY (مرة واحدة في النهاية)

**الهدف:** التحقق النهائي فقط

```
□ اختبر التدفقات الرئيسية
□ تأكد من عدم وجود regression
□ تحقق من الأمان

الناتج: نجاح / فشل (أبلغ المستخدم)
```

**ممنوع:**
- "دعني أتحقق مرة أخرى"
- إعادة الاختبار
- إضافة فحوصات جديدة

---

## PHASE 6: SYNC (مرة واحدة)

```
□ git add -A
□ git commit -m "..."
□ git push

الناتج: تم الحفظ
```

---

## 🚫 ممنوعات صارمة

| ممنوع | البديل |
|-------|--------|
| "دعني أتحقق مرة أخرى" | تحقق مرة واحدة في النهاية |
| "وجدت مشكلة جديدة" | سجّلها للجلسة القادمة |
| "أحتاج أن أراجع" | استمر أو أبلغ المستخدم |
| إعادة قراءة الملفات | اقرأ مرة في البداية فقط |
| تعديل الخطة أثناء التنفيذ | الخطة ثابتة |
| التحقق بين كل خطوة | تحقق في النهاية فقط |

---

## ✅ مسموح

- تنفيذ الخطة كما هي
- إبلاغ المستخدم بالمشاكل
- التوقف عند العقبة (لا محاولة لانهائية)
- سجل المشاكل الجديدة للجلسة القادمة

---

## مثال: تدفق صحيح

```
□ DISCOVER: خريطة النظام كاملة
□ ANALYZE: 5 مشاكل موجودة
□ PLAN: سأصلح 3 في هذه الجلسة
□ EXECUTE: خطوة 1 → خطوة 2 → خطوة 3 (بدون توقف)
□ VERIFY: اختبار نهائي واحد
□ SYNC: git push
□ DONE: أبلغ المستخدم
```

**لا Loop. لا تكرار. خطوط أنابيب واضحة.** ✅
