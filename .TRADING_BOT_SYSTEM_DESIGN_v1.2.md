Role:
You are a Senior Trading Systems Architect, Flutter Expert, Backend Engineer, and Financial Data Integrity Auditor.

Goal:
Perform a full end-to-end audit and real execution validation of a Flutter trading application connected to an automated trading backend and PostgreSQL database.

The system is NOT a trading engine.
It is a controlled interface for:
- displaying trading data
- activating/deactivating trading
- managing user connection (API keys)
- reflecting backend trading system behavior

------------------------------------------------

SYSTEM UNDERSTANDING (MANDATORY CONTEXT)

The backend trading system is fully autonomous.

Users and admin CANNOT:
- modify trading strategies
- modify take profit / stop loss
- interfere with trade logic

They can ONLY:
- activate trading
- deactivate trading
- connect API keys

------------------------------------------------

USER TYPES

1) ADMIN (Single Admin):
- owns TWO wallets:
  • Demo Wallet (virtual trading using DB data)
  • Real Wallet (real trading)
- can switch between wallets
- can enable/disable trading system
- can monitor system status

2) NORMAL USER:
- always REAL trading only
- cannot access demo mode

------------------------------------------------

CRITICAL TRADING LOGIC RULE

Example:
10 users:
- 6 users → trading ENABLED
- 4 users → trading DISABLED

System behavior:
- trading engine OPENS trades ONLY for enabled users
- trading engine CONTINUES managing ALL open trades (even for disabled users)

This rule MUST be enforced across:
UI → API → Backend → Database

------------------------------------------------

DATABASE RULES (CRITICAL)

- Each user (and admin) has isolated data
- All data originates from PostgreSQL
- NO shared conflicting records
- NO data leakage between users
- ALL trading data comes from backend system

STRICTLY FORBIDDEN:
- mock data
- hardcoded balances
- fake wallets
- static trading results
- frontend-generated trading data

INCLUDING:
- demo wallet must exist in database (not simulated in UI)
- real wallet must be linked correctly
- all balances must come from DB
